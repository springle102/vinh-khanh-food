01 • Tổng quan

Bài toán kinh doanh và mục tiêu

Hệ thống được xây dựng để số hóa trải nghiệm tham quan ẩm thực và chuẩn hóa nội dung vận hành cho Phố Ẩm thực Vĩnh Khánh.

01

Bối cảnh

Khu phố ẩm thực có nhiều quán, món đặc trưng và nhu cầu tiếp cận khách trong nước lẫn quốc tế, nhưng dữ liệu giới thiệu hiện dễ phân tán và thiếu chuẩn hóa.

Vấn đề cần giải quyết

  • Thông tin quán, món, ưu đãi và lộ trình chưa có nguồn dữ liệu thống nhất.
  • Thiếu bản dịch và audio guide nhất quán cho khách quốc tế.
  • Quản lý QR, review và media còn rời rạc, khó kiểm soát.

Mục tiêu dự án

  • Xây dựng admin dashboard tập trung.
  • Quản lý nội dung đa ngôn ngữ và audio.
  • Chuẩn hóa dữ liệu backend làm nguồn chuẩn duy nhất.

Kiến trúc tổng quan

Cấu trúc hệ thống dựa trên đồ án hiện tại

Phần này mô tả trung thực kiến trúc source đang có: một admin-web React/Vite cho vận hành, một backend-api ASP.NET Core ở phía server và một mobile-app .NET MAUI cho trải nghiệm du khách.

ARC
Kiến trúc hệ thống
Admin + Guide Mobile/Public

Đồ án hiện tại không tách thành microservice. Backend là một API duy nhất, nhưng đang phục vụ song song api/v1 cho admin web và api/guide/v1 cho mobile/public guide. Nhánh admin hiện hữu đi qua AdminDataRepository và StorageService, còn nhánh guide mới đi qua DbContext, repositories và services riêng nhưng vẫn dùng chung SQL Server và /wwwroot/storage.

ADM

admin-web

Giao diện vận hành cho SUPER_ADMIN và PLACE_OWNER, gồm dashboard, POI, content, users, promotions, reviews, activity và settings.

API

backend admin API

Phục vụ api/v1/*, bootstrap admin state, auth, CRUD và upload media cho admin web.

GDE

backend guide API

Phục vụ api/guide/v1/*, gồm mobile settings, POI list/detail, QR lookup, routes, analytics và guide admin API.

APP

mobile-app

Ứng dụng .NET MAUI cho splash, chọn ngôn ngữ, home, map, list, POI detail, QR scanner, settings và auto narration.

DB

sql-server + storage

Lưu dữ liệu dùng chung, phục vụ URL media/audio và persist các bản ghi analytics, route, translation, POI và settings.

Luồng kiến trúc hiện tại admin-web → /api/v1/* → AdminDataRepository + StorageService → SQL Server + /wwwroot/storage
mobile-app → /api/guide/v1/* → Controllers + Services + Repositories + DbContext → SQL Server + /wwwroot/storage

09 • Nhược điểm

Nhược điểm

09

Rủi ro dữ liệu

Backend hiện vẫn persist qua seed SQL thay vì cơ sở dữ liệu quan hệ hoàn chỉnh, nên khả năng mở rộng và kiểm soát đồng thời còn hạn chế.

Rủi ro nội dung

Bản dịch, audio và mô tả điểm đến cần được vận hành nhất quán; nếu quy trình kiểm duyệt lỏng, trải nghiệm du khách có thể không đồng đều.

Rủi ro phân quyền

Nếu role và phạm vi managedPlaceId không được kiểm soát chặt ở backend, người dùng có thể thao tác vượt quyền.

Phụ thuộc hạ tầng

Admin web phụ thuộc vào backend API, storage và dữ liệu seed ban đầu để có thể trình diễn đầy đủ trong môi trường đồ án.

Hướng phát triển hợp lý sau đồ án là chuyển lớp persist hiện tại sang EF Core + SQL Server nhưng vẫn giữ nguyên contract bootstrap/auth/CRUD để frontend không phải thay đổi lớn.
